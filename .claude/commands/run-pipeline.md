# Оркестратор пайплайна разработки

Ты — **главная сессия (оркестратор)**. Ты ведёшь весь цикл «разработчик → тестировщик → доработка» и сам считаешь итерации. В Claude Code суб-агент не может вызывать другого суб-агента, поэтому всю координацию делаешь ТЫ: разработчик и тестировщик только выполняют свою часть и возвращают управление тебе.

`$ARGUMENTS` — идентификатор спеки (например `SPEC-002`). Ниже `{N}` — её номер (например `002`).

Вызывай суб-агентов **явной фразой**: «Use the developer subagent: …», «Use the tester subagent: …».

## Алгоритм

### 0. Проверка спеки
- Проверь, что файл `.agent_docs/specs/$ARGUMENTS.md` существует.
- Если файла нет — покажи пользователю список доступных спек из `.agent_docs/specs/` и **остановись**.

### 1. Реализация (developer)
Вызови разработчика:

> **Use the developer subagent:** реализуй спецификацию `.agent_docs/specs/$ARGUMENTS.md` строго по инструкциям своего агента, напиши автотесты и создай отчёт `.agent_docs/dev_reports/DEV-{N}.md`. Тестировщика НЕ вызывай и проект пока НЕ поднимай — этим управляю я.

### 2. Тестирование (tester)
Вызови тестировщика:

> **Use the tester subagent:** проверь SPEC-{N} (код-ревью + автотесты) и создай `.agent_docs/test_reports/TEST-{N}.md`. Первой строкой после заголовка — вердикт **PASSED/FAILED** и номер итерации.

### 3. Чтение вердикта
Прочитай `.agent_docs/test_reports/TEST-{N}.md` и возьми вердикт из первых строк.

### 4. Цикл доработки — счётчик ведёшь ТЫ, максимум 3 итерации
Заведи счётчик итераций (первый прогон тестов = итерация 1).

- Пока вердикт **FAILED** и итераций **< 3**:
  1. Увеличь счётчик итераций.
  2. Вызови разработчика на доработку:
     > **Use the developer subagent:** прочитай `.agent_docs/test_reports/TEST-{N}.md`, исправь все пункты CRITICAL и упавшие тесты, обнови `.agent_docs/dev_reports/DEV-{N}.md` с пометкой «Итерация {i}». Тестировщика не вызывай.
  3. Повтори **шаг 2** (tester) и **шаг 3** (чтение вердикта).
- Выход из цикла: либо вердикт **PASSED**, либо достигнут лимит 3 итерации.

### 5. Если PASSED — подъём проекта для приёмки
Вызови разработчика:

> **Use the developer subagent:** подними проект для приёмки (`docker compose up -d --build`, оставь работающим, проверь что отвечает) и создай `.agent_docs/manual_checks/RUN-{N}.md`. Коммит НЕ делай.

Затем сообщи пользователю:
- вердикт **PASSED** и **сколько итераций** потребовалось;
- что проект **поднят и работает**, ждёт его **ручной приёмки**;
- адрес/точку входа для теста и путь к `RUN-{N}.md`.

### 6. Если после 3 итераций всё ещё FAILED
Сообщи пользователю оставшиеся проблемы (из последнего `TEST-{N}.md`) и **остановись**. Проект не поднимай, коммит не делай.

### 7. Коммит — отдельно, после «ок» пользователя
Сам в пайплайне коммит **не делай**. Когда пользователь подтвердит приёмку («ок») — вызови разработчика отдельной командой:

> **Use the developer subagent:** оформи git-коммит по $ARGUMENTS в стиле Conventional Commits (`Ref: SPEC-{N}`). `git push` и деплой — только если пользователь попросит это отдельно.

## Правила
- Цикл и счётчик из 3 итераций — на тебе (оркестраторе), не на разработчике.
- Суб-агенты не вызывают друг друга; каждый шаг инициируешь ты.
- Без явного «ок»/команды пользователя — не коммить, не пушить, не деплоить.
