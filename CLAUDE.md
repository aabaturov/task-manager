# Проект — Инструкции для агентов

## Система агентов

1. **Аналитик** — обсуждает фичи в Chat (отдельный проект), формирует спецификации
2. **Оркестратор** — главная сессия (команда `/run-pipeline`). Ведёт цикл
   «разработчик → тестировщик → доработка» и считает итерации.
3. **Разработчик** — реализует фичи по спецификации, готовит локальный запуск
4. **Тестировщик** — код-ревью + автотесты (автоматические проверки)

> В Claude Code суб-агент не может вызывать другого суб-агента (вложенность
> запрещена). Поэтому координацию (вызовы агентов, цикл, счётчик итераций)
> ведёт **только оркестратор**; разработчик и тестировщик друг друга не вызывают.

## Выбор стека и инструментов

- Если стек в проекте уже задан — строго следуй ему.
- Если проект пустой — выбери разумные мейнстрим-дефолты под задачу.
- Используй **актуальные инструменты, которыми пользуется большинство**. НЕ тащи экспериментальное/bleeding-edge и НЕ бери устаревшее/deprecated.
- Зафиксируй выбранный стек в этом файле, чтобы следующие итерации его придерживались.

## Директория обмена между агентами

```
.agent_docs/
├── specs/           # SPEC-{N}.md   — от аналитика
├── dev_reports/     # DEV-{N}.md    — от разработчика
├── test_reports/    # TEST-{N}.md   — от тестировщика
├── manual_checks/   # RUN-{N}.md    — чек-лист для ручного теста пользователем
├── bugfixes/        # BUGFIX-{N}.md
└── templates/
```

## Рабочий процесс

Циклом и счётчиком из 3 итераций управляет **главная сессия (оркестратор —
команда `/run-pipeline`)**, а НЕ разработчик. Суб-агенты друг друга не вызывают.

1. Кладёшь `SPEC-{N}.md` в `.agent_docs/specs/`
2. Запускаешь `/run-pipeline SPEC-{N}` — дальше всё координирует оркестратор:
3. Оркестратор вызывает **разработчика** → реализация + тесты + `DEV-{N}.md`
4. Оркестратор вызывает **тестировщика** → `TEST-{N}.md` с вердиктом PASSED/FAILED
5. Если **FAILED** — оркестратор снова вызывает разработчика на доработку
   (передаёт ему `TEST-{N}.md`), затем снова тестировщика. **Максимум 3 итерации**;
   счётчик ведёт оркестратор, а не разработчик.
6. Когда **PASSED** — оркестратор просит разработчика поднять проект
   (`docker compose up -d`) и **оставить работающим**, написать `RUN-{N}.md`
7. **Приёмка — за тобой:** тестируешь руками на поднятом проекте и даёшь добро.
   После твоего «ок» оркестратор отдельно вызывает разработчика для git-коммита
   (Conventional Commits). `git push` и деплой на сервер — только по отдельной
   явной команде

## Нумерация

Номер спеки связывает все документы: SPEC-003 → DEV-003 → TEST-003 → RUN-003.
