# Проект — Инструкции для агентов

## Система агентов

1. **Аналитик** — обсуждает фичи в Chat (отдельный проект), формирует спецификации
2. **Разработчик** — реализует фичи по спецификации, готовит локальный запуск
3. **Тестировщик** — код-ревью + автотесты (автоматические проверки)

## Выбор стека и инструментов

- Если стек в проекте уже задан — строго следуй ему.
- Если проект пустой — выбери разумные мейнстрим-дефолты под задачу.
- Используй **актуальные инструменты, которыми пользуется большинство**. НЕ тащи экспериментальное/bleeding-edge и НЕ бери устаревшее/deprecated.
- Зафиксируй выбранный стек в этом файле, чтобы следующие итерации его придерживались.

## Директория обмена между агентами

```
.agent_docs/
├── specs/           # SPEC-{N}.md   — от аналитика
├── dev_reports/     # DEV-{N}.md    — от разработчика
├── test_reports/    # TEST-{N}.md   — от тестировщика
├── manual_checks/   # RUN-{N}.md    — чек-лист для ручного теста пользователем
├── bugfixes/        # BUGFIX-{N}.md
└── templates/
```

## Рабочий процесс

1. Кладёшь `SPEC-{N}.md` в `.agent_docs/specs/`
2. Запускаешь `/run-pipeline SPEC-{N}`
3. Разработчик реализует фичи + пишет тесты, создаёт `DEV-{N}.md`
4. Тестировщик гоняет автопроверки, создаёт `TEST-{N}.md` с вердиктом PASSED/FAILED
5. Если FAILED — разработчик фиксит. **Максимум 3 итерации фикс→тест.**
6. Когда вердикт PASSED — разработчик сам поднимает проект локально (`docker compose up -d`) и **оставляет работающим**, пишет `RUN-{N}.md`
7. **Приёмка — за тобой:** тестируешь руками на поднятом проекте и даёшь добро. После твоего «ок» разработчик оформляет git-коммит с осмысленным сообщением (Conventional Commits). `git push` и деплой на сервер — только по отдельной явной команде

## Нумерация

Номер спеки связывает все документы: SPEC-003 → DEV-003 → TEST-003 → RUN-003.
