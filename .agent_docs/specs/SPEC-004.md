# Спецификация — SPEC-002

**Дата:** 28.06.2026
**Статус:** Утверждено

## Обзор

Пакет перестраивает верхнюю зону интерфейса в три именованных блока (**Дела на день**, **Лёгкие дела (не забыть)**, **Календарь**), добавляет два новых блока с их сущностями и переименовывает группировку проектов. Главное новое понятие — **событие**: дело с привязкой к конкретной дате (и времени), которое показывается в календаре-сетке. Событие может быть самостоятельным или задачей проекта. Создание событий и просмотр «лёгких дел» доступны в том числе из Telegram-бота; бот напоминает о событиях накануне. Пакет опирается на v1.0 и SPEC-001 и частично заменяет визуальную раскладку верхней зоны из SPEC-001 (см. «Зависимости»).

---

## Фичи

### Фича 1: Перестройка верхней зоны и доски проектов

**Описание:** Верх интерфейса (веб) — три блока в ряд: «Дела на день» (поведение из SPEC-001 без изменений), «Лёгкие дела (не забыть)» (Фича 2) и «Календарь» (Фича 3). Ниже — доска проектов. Группировка проектов переименована: **временные** (бывшие «локальные», разовые/проектные) и **постоянные** (бывшие «глобальные», сферы вроде Спорта) — та же ось, новые названия, поведение прежнее. Эта раскладка заменяет «закреплённую зону» из SPEC-001; панель «Дела на день» теперь живёт как один из трёх блоков.

**Сценарии:**
1. Пользователь открывает веб → видит сверху три блока в ряд (День / Лёгкие дела / Календарь), ниже — доску проектов, разбитую на «временные», затем «постоянные».
2. Пользователь смотрит на блок «Дела на день» → он визуально выделен (жёлтый); два других блока — в обычной схеме.
3. Пользователь меняет тип проекта временный↔постоянный → проект переходит в соответствующую группу доски.

**Граничные случаи:**
- Жёлтым выделяется только блок «Дела на день»; «Лёгкие дела» и «Календарь» — обычная цветовая схема.
- На телефоне три блока перестраиваются в столбик (адаптив), доска — ниже.
- Если в доске нет временных или нет постоянных проектов → пустая группа не показывается.
- Тип проекта по умолчанию при создании — **временный** (бывший «локальный»).
- Закреплённые проекты (из SPEC-001) сохраняются как возможность и показываются отдельной группой «закреплённые» над «временными»/«постоянными» (см. заметку для разработчика — этот пункт помечен на подтверждение).
- Существующие проекты после миграции: «локальный» → «временный», «глобальный» → «постоянный», без потери данных.

**Критерии приёмки:**
- [ ] Сверху три блока в ряд: «Дела на день», «Лёгкие дела (не забыть)», «Календарь».
- [ ] Жёлтым выделён только блок «Дела на день».
- [ ] Доска проектов разбита на «временные» и «постоянные»; новый проект по умолчанию временный.
- [ ] На телефоне блоки корректно перестраиваются в столбик.
- [ ] Поведение панели «Дела на день» не изменилось относительно SPEC-001.

### Фича 2: Лёгкие дела (не забыть)

**Описание:** Блок-список мелочей, которые надо не потерять. Это **самостоятельный список** — свои записи, не связаны с проектами и их задачами. Не пересекается с пометкой «важное» из SPEC-001 (это разные вещи). Создание и управление — в вебе; бот только присылает список на просмотр (Фича 4).

**Сценарии:**
1. Пользователь вводит текст в блоке «Лёгкие дела» → запись добавляется в конец списка.
2. Пользователь отмечает лёгкое дело сделанным → оно зачёркивается, остаётся в списке.
3. Пользователь удаляет лёгкое дело → оно исчезает из списка.

**Граничные случаи:**
- Если текст пустой или из одних пробелов → запись не добавляется.
- Лишние пробелы по краям текста обрезаются.
- Порядок записей — по времени добавления (новые в конец); ручной сортировки в этой версии нет.
- Если список пуст → блок показывает «пусто» / «нет дел».
- Лёгкие дела не отображаются на доске проектов, в панели «Дела на день» и в календаре — только в своём блоке (и в боте на просмотр).
- Создание лёгких дел — только в вебе; через бота их создавать нельзя (бот только просмотр).

**Критерии приёмки:**
- [ ] В блоке можно добавить лёгкое дело (непустой текст).
- [ ] Лёгкое дело можно отметить сделанным (зачёркивание) и удалить.
- [ ] Записи идут по порядку добавления; пустой список показывает заглушку.
- [ ] Список независим от задач проектов и от пометки «важное».

### Фича 3: Календарь и события

**Описание:** Отдельная утилита-календарь (веб): сетка дат квадратиками, как в гугл-календаре. В клетках показываются **события** — дела, привязанные к конкретной дате. **Событие** — это дело с привязкой к дате и времени; обычное дело без даты остаётся просто делом. Событие может быть **самостоятельным** (не принадлежит проекту) или **задачей проекта**, которой добавили дату. Типов у событий нет — только текст, дата, время.

**Важно:** привязка к дате ≠ дедлайн. Дата — день, когда дело надо непосредственно сделать. Дедлайн (SPEC-001) — крайний срок выполнения; это отдельное поле, в календарь оно **не** попадает. У одной задачи проекта могут быть и дедлайн, и привязка к дате — независимо.

**Сценарии:**
1. Пользователь открывает «Календарь» → видит сетку дней с вписанными событиями на их датах.
2. Пользователь кликает по дню → форма создания (текст + время) → создаётся самостоятельное событие на этот день, появляется в клетке.
3. В клетке дня много событий, не помещаются → пользователь кликает по дню → день раскрывается со полным списком событий.
4. Событие-задача проекта помечается сделанным (в проекте или в панели) → в календаре оно показывается зачёркнутым.
5. Пользователь листает месяцы вперёд/назад → видит будущие и прошедшие события на их датах.

**Граничные случаи:**
- Время у события **обязательно** (v1): создание без времени отклоняется.
- Создание события в календаре даёт **самостоятельное** событие (без проекта). Привязка к дате уже существующей задачи проекта делается со стороны задачи/бота (Фича 4), а не из календаря.
- Прошедшие события остаются на своей дате (видны при листании назад); автоархива и скрытия нет.
- Если задача проекта с привязкой к дате удаляется → событие исчезает из календаря.
- Если у задачи проекта снять привязку к дате → она пропадает из календаря, оставаясь обычной задачей.
- Редактирование текста задачи/события → отражается в календаре.
- Самостоятельное событие можно отметить сделанным (зачёркивается) и удалить прямо в календаре; на доске проектов оно не появляется (живёт только в календаре).
- Несколько событий на один день/время допускаются; конфликтов система не предотвращает.
- Дата интерпретируется в едином часовом поясе приложения (из `.env`) — согласовано с напоминаниями бота.
- Самостоятельные события не попадают в слоты панели «Дела на день» (панель ссылается только на задачи проектов, как в SPEC-001).

**Критерии приёмки:**
- [ ] Календарь показывает сетку дат с событиями в клетках на их датах.
- [ ] Событие можно создать в календаре кликом по дню: текст + время (время обязательно).
- [ ] Переполненная клетка раскрывается по клику в полный список дня.
- [ ] Привязка к дате и дедлайн — разные поля; дедлайн в календаре не показывается.
- [ ] Прошедшие события остаются на своих датах.
- [ ] Событие-задача проекта отражает в календаре правки текста и статус «сделано».

### Фича 4: Бот — события и просмотр «не забыть»

**Описание:** Telegram-бот получает: (а) шаг привязки к дате при добавлении дела — чтобы создавать события; (б) кнопку «прислать дела „не забыть"» (только просмотр); (в) напоминание о событиях накануне. Доступ — только владельцу (как в v1.0).

**Сценарии:**
1. Пользователь пишет боту текст дела → бот предлагает: «привязать к дате» либо выбрать проект.
   - Выбирает «привязать к дате» → бот запрашивает дату, затем время → создаётся самостоятельное событие, появляется в календаре.
   - Выбирает проект → после выбора проекта бот спрашивает «привязать к дате?» → если да, запрашивает дату и время → задача создаётся в проекте и получает привязку к дате (показывается в календаре); если нет → обычная задача проекта, как в v1.0.
2. Пользователь жмёт кнопку «прислать „не забыть"» → бот присылает текущий список лёгких дел одним сообщением (только просмотр).
3. Накануне события → бот присылает напоминание со списком событий на следующий день.

**Граничные случаи:**
- Время при создании события через бота обязательно; если введено пусто/некорректно → бот переспрашивает.
- Некорректная дата → бот переспрашивает, событие не создаётся.
- Создание события на прошедшую дату разрешено (попадёт в прошлый день календаря); напоминание для прошедшей даты не шлётся.
- Кнопка «не забыть» при пустом списке → бот сообщает, что список пуст.
- Из сообщения «не забыть» нельзя отмечать/удалять — только просмотр.
- **Напоминание накануне:** вечером предыдущего дня в фиксированное время из `.env`; одно напоминание на день (списком всех событий назавтра); повторного напоминания в момент события в v1 нет.
- Событие создано на **сегодня** → «накануне» уже прошло, напоминание не шлётся (ограничение v1).
- **Бот лежал в момент напоминания:** при старте досылает напоминания, чьё время приходится на сегодня; более старые пропускает.
- Часовой пояс для дат и времени напоминаний — единый, задаётся в `.env`.

**Критерии приёмки:**
- [ ] При добавлении дела бот даёт выбор «привязать к дате» или проект; ветка даты создаёт событие с датой и временем.
- [ ] Для задачи проекта бот дополнительно спрашивает про привязку к дате; при «да» задача появляется в календаре.
- [ ] Кнопка «прислать „не забыть"» присылает список лёгких дел (просмотр, без действий).
- [ ] Накануне бот присылает напоминание со событиями на следующий день.
- [ ] После простоя бот досылает сегодняшние пропущенные напоминания и не шлёт устаревшие.

## Технические требования

- **Сущность «событие» (привязка к дате):** поля — текст, дата, время (обязательно в v1), статус «сделано». Событие либо самостоятельное (без проекта), либо это задача проекта с заданными датой и временем. Конкретная реализация хранения на усмотрение разработчика (например, задача с пустым `project_id` и полями даты/времени, либо отдельная таблица событий) — обязательное требование: самостоятельное событие без проекта допустимо, а задача проекта может нести дату/время.
- **Привязка к дате ≠ дедлайн:** это разные поля с разным смыслом; дедлайн (SPEC-001) в календаре не участвует. Оба могут присутствовать одновременно.
- **Лёгкие дела:** отдельный список (текст, статус «сделано», время добавления, порядок). Не связаны с проектами, задачами и «важным».
- **Проекты:** значения/подписи типа переименовать «локальный»→«временный», «глобальный»→«постоянный»; значение по умолчанию — «временный». Миграция существующих проектов.
- **Календарь:** веб (FastAPI отдаёт данные, React рендерит сетку); API для чтения/создания/изменения событий по датам.
- **Бот:** новый шаг привязки к дате в потоке добавления; кнопка просмотра «не забыть»; планировщик напоминаний накануне; досыл пропущенных при старте.
- **Часовой пояс:** единый, из `.env` (для интерпретации дат и времени напоминаний).
- **Совместимость:** существующие задачи без даты в календаре не появляются; данные SPEC-001 (важное, дедлайн, done, порядок, закрепление) сохраняются.

## Зависимости

- Базируется на v1.0 (проекты → задачи, авторизация, бот, SQLite) и SPEC-001 (статус «сделано», редактирование задач, организация доски, панель «Дела на день»).
- **Заменяет часть SPEC-001:** «закреплённая зона» (Фичи 6 и 8 SPEC-001) уступает место трёхблочной верхней зоне; панель «Дела на день» переезжает в один из трёх блоков, её внутреннее поведение не меняется. Это намеренная эволюция, а не противоречие; SPEC-001 при этом не редактируется — изменение вносит разработчик в рамках SPEC-002.
- Календарь переиспользует статус «сделано» и редактирование задач из SPEC-001 для отображения событий-задач.

## Заметки для разработчика

- **Типов у событий нет** (сознательно убрано на этапе проектирования): только текст, дата, время.
- **Привязка к дате — это «когда делаю», дедлайн — «крайний срок».** Не смешивать; календарь питается только привязкой к дате.
- **Закреплённые проекты в новой раскладке** — заложен дефолт «отдельная группа над временными/постоянными». Это единственный пункт, помеченный на подтверждение заказчиком: возможна и другая трактовка (всплытие наверх своей группы или отказ от закрепления). Уточнить до реализации этого участка.
- Панель «Дела на день» из SPEC-001 переносится без изменений в поведении; меняется только её положение (один из трёх блоков) — жёлтый фон сохраняется.
- Лёгкие дела создаются только в вебе; бот по ним — просмотр.
- В v1 нет напоминания в момент события и нет напоминания для событий, созданных на сегодня — только «накануне».
- Версионирование по правилам проекта: это новая функциональность → мажор `3.0`; обновить changelog в README.
