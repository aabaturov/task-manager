# Спецификация — SPEC-002

**Дата:** 21 июня 2026
**Статус:** Утверждено

## Обзор
Доработка существующего Telegram-бота менеджера задач (python-telegram-bot 22.x, общая SQLite-база с вебом). Две фичи: постоянная reply-клавиатура вместо текстовых команд и новый экран только для просмотра «Дела на день» (слоты). Бот читает тот же источник данных, что веб (`DayPanel`); схема БД не меняется. Доступ — только у владельца.

## Фичи

### Фича 1: Постоянная клавиатура

**Описание:** На `/start` бот шлёт приветствие и постоянную reply-клавиатуру с двумя кнопками, открывающими экран всех задач и экран дел на день. Произвольный текст по-прежнему создаёт задачу.

**Сценарии:**
1. Владелец → `/start` → приветствие + `ReplyKeyboardMarkup(resize_keyboard=True, is_persistent=True)` с кнопками `📋 Все задачи` и `📅 Дела на день` в один ряд.
2. Нажатие `📋 Все задачи` → существующая логика `/tasks`.
3. Нажатие `📅 Дела на день` → экран из Фичи 2.
4. Произвольный текст, не равный подписям кнопок → текущий поток добавления задачи (inline-кнопки выбора проекта → сохранение).
5. Команды `/tasks` и `/start` продолжают работать.

**Граничные случаи:**
- Текст сообщения точно совпадает с подписью кнопки → трактуется как нажатие, не как новая задача. Следствие: строки `📋 Все задачи` и `📅 Дела на день` нельзя сохранить как текст задачи — осознанный трейдофф для персонального инструмента.
- Обработчики кнопок зарегистрированы после общего обработчика добавления → баг «создаётся задача с названием кнопки». Порядок строгий: сначала точные обработчики кнопок (`filters.Text(["📋 Все задачи", "📅 Дела на день"])`), затем `filters.TEXT & ~filters.COMMAND` — последним.
- `/start` или нажатие пришло не от владельца (≠ `TELEGRAM_ALLOWED_USER_ID`) → отказ/игнор как сейчас; клавиатура и данные не показываются.
- Старый клиент не рисует reply-клавиатуру → fallback: `/tasks`, `/start` и набор текста работают; по желанию `set_my_commands` для меню команд.
- Повторный `/start` → повторно шлёт клавиатуру, задач не создаёт (идемпотентно).
- Подписи кнопок и строки в фильтрах — одна и та же константа (эмодзи матчатся посимвольно).

**Критерии приёмки:**
- [ ] `/start` от владельца показывает приветствие и клавиатуру с двумя кнопками в один ряд.
- [ ] `📋 Все задачи` даёт тот же вывод, что `/tasks`.
- [ ] `📅 Дела на день` открывает экран Фичи 2.
- [ ] Произвольный текст по-прежнему создаёт задачу через выбор проекта.
- [ ] Нажатие любой из двух кнопок не создаёт задачу.
- [ ] Не-владелец не получает ни клавиатуру, ни данные.
- [ ] Бот стартует и импортируется без ошибок.

### Фича 2: Экран «Дела на день» (только просмотр)

**Описание:** Новый экран читает слоты из того же источника, что веб (`DayPanel`). Слот = `index` + список `task_ids`. Сообщение формируется по непустым слотам в порядке `index`. Отметку «выполнено» из чата не добавляем.

**Сценарии:**
1. Есть непустые слоты → для каждого (в порядке `index`): заголовок `Слот N` + список задач. Каждая задача — подпись её проекта + текст. Выполненные (`done`) выводятся как `✅` + зачёркивание.
2. Слотов нет или все пусты → одно сообщение «На день пока ничего не выбрано».

**Формат вывода:**
- `parse_mode = HTML`.
- Текст задачи и название проекта **экранируются** (минимум `<`, `>`, `&`) перед вставкой в сообщение.
- Выполненная задача: `✅ <s>текст</s>`. Невыполненная: текст без пометки.
- Нумерация заголовков — как в вебе, 1-based (`Слот 1`, `Слот 2`, …). Если в БД `index` 0-based — выводить `index + 1`.

**Граничные случаи:**
- Пустой слот (`task_ids` пуст) → пропускается, заголовок не выводится.
- Висячих `task_id` не возникает: веб подчищает слоты при удалении задач/проектов (вариант (в), подтверждён владельцем). **Защитный fallback:** если при чтении `id` всё же не резолвится в задачу, бот молча пропускает его и не падает; если слот из-за этого опустел — пропускает и слот.
- Сообщение длиннее лимита Telegram (4096 символов) при большом числе задач → разбивать на несколько сообщений по границам слотов (не рвать слот посередине).
- Конкурентность: веб меняет слоты во время чтения → бот отдаёт снимок на момент запроса; для просмотра достаточно одного запроса, блокировки не нужны.

**Критерии приёмки:**
- [ ] Экран читает слоты из того же источника, что `DayPanel`; новых таблиц/полей не вводит.
- [ ] Непустые слоты выводятся по порядку `index`, заголовок `Слот N` 1-based.
- [ ] Каждая задача показывает подпись проекта и текст.
- [ ] Выполненные помечены `✅` + зачёркиванием; сообщение отправляется без сбоя на спецсимволах (экранирование работает).
- [ ] Пустые слоты не выводятся.
- [ ] Ничего не выбрано → сообщение «На день пока ничего не выбрано».
- [ ] Экран только для просмотра: кнопок отметки «выполнено» нет.
- [ ] Висячий `id` (если возник) не роняет экран — пропускается.
- [ ] Длинный список не вызывает ошибку Telegram.

## Технические требования
- python-telegram-bot 22.x; бота не пересобирать, дописывать поверх текущего кода.
- Источник слотов — существующая модель/способ чтения, что у `DayPanel`; схема БД не меняется.
- Проверка `TELEGRAM_ALLOWED_USER_ID` сохраняется на всех новых точках входа (кнопки и новый экран).
- Порядок регистрации хендлеров: точные обработчики кнопок → общий обработчик добавления задачи последним.
- Подписи кнопок и строки фильтров — единые константы.
- `parse_mode = HTML` с экранированием `< > &` для пользовательского текста (тексты задач, названия проектов).

## Зависимости
