## Реализованные фичи

### Фича 1 — Перестройка верхней зоны и доски проектов
- Верх веба — три именованных блока в ряд: **Дела на день** (DayPanel, без
  изменений поведения), **Лёгкие дела (не забыть)**, **Календарь**
  (`frontend/src/Board.jsx`, контейнер `.top-blocks`).
- Жёлтым выделён только блок «Дела на день»: цвет вынесен в CSS-переменную
  `--day-bg` (теперь `#fff8e1`), которой питается `.day-panel`. Остальные два
  блока — обычная светлая схема (`.block`).
- Доска проектов ниже разбита на группы: **Закреплённые → Временные →
  Постоянные**; пустые группы не рендерятся; заголовки групп показываются,
  только когда непустых групп больше одной.
- Тип проекта переименован на уровне значений: `local → temporary`,
  `global → permanent`; дефолт при создании — `temporary`
  (`schemas.py` Literal, `models.py` default, `Board.addProject`,
  `ProjectPanel` editor — подписи «Временный/Постоянный»).
- Адаптив: на ширине ≤980px три блока перестраиваются в столбик, доска ниже.

### Фича 2 — Лёгкие дела (не забыть)
- Отдельная таблица `light_tasks` (текст, done, created_at, position) —
  не связана с проектами/задачами/«важным».
- API: `GET/POST/PATCH/DELETE /api/light-tasks` (`main.py`). Пустой/пробельный
  текст отклоняется (422), пробелы по краям обрезаются, порядок — по времени
  добавления.
- UI: `frontend/src/LightTasksPanel.jsx` — добавление, отметка «сделано»
  (зачёркивание, запись остаётся), удаление, заглушка «Нет дел» при пустом.

### Фича 3 — Календарь и события
- «Событие» = задача с привязкой к дате и времени. Хранение: поля
  `tasks.event_date` / `tasks.event_time`; `tasks.project_id` сделан
  **nullable** (самостоятельное событие — задача без проекта).
- Привязка к дате независима от `deadline` (отдельные поля; календарь питается
  только `event_date`).
- API:
  - `GET /api/events?start=&end=` — события в диапазоне дат (сортировка
    дата+время), источник для сетки и для напоминаний бота.
  - `POST /api/events` — создать самостоятельное событие (текст + дата + время;
    время обязательно — на уровне схемы).
  - `PATCH/DELETE /api/events/{id}` — только для самостоятельных событий;
    попытка применить к задаче проекта → 400. Очистка времени запрещена (422).
  - `PUT /api/tasks/{id}/event` — привязать/снять дату-время у задачи проекта
    (оба поля задаются/снимаются вместе; иначе 422).
- UI: `frontend/src/CalendarPanel.jsx` — сетка месяца (Mon-first), события в
  клетках, перелистывание месяцев, «Сегодня». Клик по дню → модалка: полный
  список событий дня + форма создания самостоятельного события (текст +
  обязательное время). Выполненные события зачёркнуты; самостоятельные можно
  отметить done и удалить прямо в дне; события-задачи проекта помечать в дне
  нельзя (отметка идёт со стороны проекта/панели) — отражаются текст и статус
  done. Хелперы сетки/времени — `frontend/src/helpers.js`.

### Фича 4 — Бот: события и просмотр «не забыть»
- Поток добавления (`bot.py`): на текст дела бот предлагает «📅 Привязать к
  дате» **или** выбрать проект.
  - Ветка даты → запрос даты, затем времени (оба переспрашиваются при ошибке) →
    создаётся самостоятельное событие.
  - Ветка проекта → задача сохраняется в проект, затем вопрос «Привязать к
    дате?»; «да» → дата+время → задача получает привязку (видна в календаре);
    «нет» → обычная задача проекта (как в v1.0).
- Кнопка **«📝 Не забыть»** — присылает список лёгких дел (только просмотр;
  без действий), при пустом списке — сообщение «пусто».
- Напоминания: планировщик (PTB JobQueue) шлёт напоминание накануне в
  `REMINDER_TIME` (часовой пояс `APP_TIMEZONE`); при старте, если время на
  сегодня уже прошло — досылает сегодняшнее напоминание один раз, более старые
  пропускает. Парсеры даты/времени переспрашивают при некорректном вводе.
- Reply-клавиатура расширена третьей кнопкой (ряд 2: «📝 Не забыть»).

## Известные ограничения и открытые вопросы

1. **Закреплённые проекты в новой раскладке — НА ПОДТВЕРЖДЕНИЕ ЗАКАЗЧИКА.**
   Спека помечает этот пункт на уточнение и задаёт дефолт: «закреплённые —
   отдельной группой над временными/постоянными». Реализован именно дефолт
   (группа «Закреплённые» первой в `normal-board`). Возможна другая трактовка
   (всплытие наверх своей группы / отказ от закрепления) — **требуется решение
   пользователя**; механику закрепления из SPEC-001 не меняли.
2. **Жёлтый цвет блока «Дела на день» vs SPEC-003.** SPEC-003 сделал панель
   белой (приподнятая карточка). SPEC-004 требует «жёлтый только у этого
   блока». Выполнено через значение переменной `--day-bg`, чтобы не ломать
   статические тесты SPEC-003. Если важнее визуал SPEC-003 — это конфликт двух
   спек, выносится на решение пользователя.
2-bis. Версия README до пакета уже стояла `3.0` под SPEC-002. По правилу
   «новая функциональность → мажор» подняли до `4.0`.
3. **Напоминания v1 (как в спеке):** нет напоминания в момент события; для
   событий, созданных на сегодня, «накануне» уже прошло — напоминание не шлётся;
   при простое досылается только сегодняшнее пропущенное, старые пропускаются.
4. **JobQueue требует extra.** Для работы напоминаний в Docker добавлен
   `python-telegram-bot[job-queue]` (APScheduler). В тестовом venv extra не
   установлен — это не влияет на тесты (JobQueue инстанцируется только в
   `bot.main()`, который тесты не вызывают). При первом `docker compose build`
   зависимость подтянется.
5. **Фронтенд не собирался (vite).** JSX написан по образцу существующих
   компонентов; визуальную сборку/рендер проверит отдельный шаг.
6. **Редактирование текста/времени самостоятельного события из UI** доступно
   как API (`PATCH /api/events/{id}`), но в `CalendarPanel` из правок дня
   реализованы отметка done и удаление + создание; инлайн-правка текста/времени
   существующего события в модалке не добавлена (спека явно требует создание,
   отметку done и удаление — они есть; правка текста события отнесена к задаче
   проекта со стороны задачи). Если нужна правка standalone-события из
   календаря — добавим по запросу.
